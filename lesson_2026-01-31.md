## ❌ 발생한 문제들

### 문제 1: `removeChild` 에러
```
Error: Failed to execute 'removeChild' on 'Node': The node to be removed is not a child of this node.
```

**원인**: `getElementsByTagNameNS`는 모든 하위 요소를 검색하므로, 찾은 요소가 `rPr`의 직접 자식이 아닐 수 있음

**시도한 해결책**: `parentNode === rPr` 체크 추가
```typescript
if (existingSolidFill && existingSolidFill.parentNode === rPr) {
    rPr.removeChild(existingSolidFill);
}
```

### 문제 2: 스타일이 PowerPoint에서 인식 안됨
**증상**: XML에는 스타일 데이터가 있지만 PowerPoint에서 표시되지 않음

**분석 결과**:
| 항목 | 원본 | 이전 번역 | 새 번역 |
|------|------|-----------|---------|
| Underlines | 41 | 1 | 31 |
| Blue Color | 26 | 13 | 23 |
| Highlight | 2 | 0 | 19 |

→ 새 번역 파일에 스타일 **데이터는 존재**하지만 인식되지 않음

**추정 원인**: 
- PowerPoint XML은 요소 순서에 민감함
- `<a:rPr>` 내부 요소들의 배치 순서가 잘못되었을 가능성
- 예: `<a:solidFill>`, `<a:highlight>`, `<a:latin>` 등의 순서

## 📚 교훈 (Lessons Learned)

### 1. PowerPoint XML 구조의 복잡성
- 단순히 요소를 추가한다고 작동하지 않음
- **요소 순서**, **네임스페이스**, **속성 형식**이 모두 중요

### 2. 점진적 접근 필요
- 한 번에 여러 기능(밑줄+색상+하이라이트)을 추가하지 말 것
- **하나씩 추가하고 테스트**하는 방식이 더 안전

### 3. 디버깅 방법
- Python `zipfile`로 PPTX 내부 XML 직접 분석
- 원본 파일과 생성된 파일의 XML 구조 비교

### 4. AI 프롬프트 vs 코드 처리
- 복잡한 스타일 보존은 **프롬프트만으로는 해결 어려움**
- 코드 레벨에서 정확한 처리 필요
